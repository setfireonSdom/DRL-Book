2  Reinforcement Learning Overview

Reinforcement learning is a framework for learning through interaction. An agent observes a situation, chooses an action, receives a reward, and moves to a new situation. Over time, it tries to improve its decisions so that the total reward it collects becomes as large as possible.

This description sounds simple, but it hides an important difference from other machine learning settings. In supervised learning, the correct answer is usually available in the training data. In reinforcement learning, the agent often has to discover good behavior by trial and error. It must balance two conflicting goals:

  1. It must use what it already knows to gain reward.
  2. It must explore actions whose value is still uncertain.

That tension between exploitation and exploration is one of the defining features of the field.

2.1 Sequential Decision-Making

The central object in reinforcement learning is not a single prediction but a sequence of decisions. A good action is not simply one that gives immediate reward. It is an action that contributes to good long-term outcomes.

This is why the same action can be good in one context and bad in another. For example, in a game, sacrificing a piece may look costly in the short term but lead to a winning position later. In navigation, taking a longer route may avoid a dead end. In robotics, a stabilizing action may temporarily reduce speed while improving future control.

Reinforcement learning formalizes these problems by focusing on trajectories:

\[ S_0, A_0, R_1, S_1, A_1, R_2, S_2, \dots \]

At each time step, the agent is in a state \(S_t\), chooses an action \(A_t\), receives a reward \(R_{t+1}\), and transitions to a new state \(S_{t+1}\). The objective is to choose actions so that the long-term return is large.

2.2 The Agent and the Environment

The agent is the learner or decision-maker. The environment is everything outside the agent that responds to its actions. This distinction is conceptual rather than physical. In a board game, the environment includes the game rules and board configuration. In robotic control, the environment may include the robot body, the external world, sensors, and dynamics not directly controlled by the learner.

The interaction loop usually looks like this:

  1. The environment presents a state to the agent.
  2. The agent chooses an action according to its current policy.
  3. The environment returns a reward and a next state.
  4. The agent updates its internal estimates or policy.

This loop may happen a few times in a toy problem or millions of times in a large-scale training run.

2.3 States, Actions, and Rewards

A state is the information used to describe the current situation. An action is a choice available to the agent. A reward is a scalar signal that evaluates the immediate outcome of an action.

It is important not to confuse reward with objective. Reward is the training signal provided at each step. The objective is usually the expected sum of rewards over time. A reward can be sparse, delayed, shaped, noisy, or even misleading. Much of reinforcement learning becomes difficult when the reward does not immediately reveal which behavior is truly good.

The quality of the state representation also matters. If the state omits critical information, the problem becomes harder because the agent cannot tell apart situations that require different behavior. This is one motivation for later deep reinforcement learning methods, where learned representations become part of the solution.

2.4 Policies and Value Functions

A policy tells the agent how to act. We often write a policy as \(\pi(a \mid s)\), the probability of choosing action \(a\) in state \(s\). A deterministic policy always chooses the same action for a given state. A stochastic policy defines a distribution over actions.

While a policy specifies behavior, a value function measures how good states or actions are under a policy. Two common value functions are:

  1. The state-value function \(V^\pi(s)\), which measures the expected return starting from state \(s\) and following policy \(\pi\).
  2. The action-value function \(Q^\pi(s, a)\), which measures the expected return after taking action \(a\) in state \(s\) and then following policy \(\pi\).

These objects are central because many algorithms either estimate value functions directly or use them to improve a policy.

2.5 Return and Discounting

The return is the total reward accumulated over time. A common definition is the discounted return:

\[ G_t = R_{t+1} + \gamma R_{t+2} + \gamma^2 R_{t+3} + \cdots \]

where \(0 \le \gamma \le 1\) is the discount factor.

The discount factor controls how much the agent cares about the future. A small \(\gamma\) makes the agent short-sighted. A large \(\gamma\) places more weight on long-term consequences. Discounting is useful for both mathematical and practical reasons. It can ensure that infinite sums remain finite, and it can encode a preference for earlier rewards over later ones.

2.6 Episodes and Continuing Tasks

Some reinforcement learning problems have natural endings. A game ends when one player wins. A robot navigation task may end when the goal is reached. These are episodic tasks.

Other problems continue indefinitely, such as recommendation systems or certain control problems. These are often treated as continuing tasks. The difference matters because it affects how we define returns, evaluate performance, and structure training.

2.7 Model-Based and Model-Free Learning

An agent is model-based if it has access to, or learns, a model of the environment dynamics. That means it can predict how actions change states and rewards. An agent is model-free if it learns behavior or value estimates directly from experience without explicitly learning the transition model.

This book begins mostly with model-free learning, because that path leads naturally into modern deep reinforcement learning. Still, model-based ideas remain important, especially when sample efficiency matters.

2.8 Why Reinforcement Learning Matters

Reinforcement learning matters because many real-world problems involve decisions, feedback, uncertainty, and long-term consequences. The framework gives us a language for these problems even when perfect solutions are impossible.

At the same time, reinforcement learning should not be treated as a universal hammer. It is often hard to train, expensive in data, and sensitive to problem design. A major part of real expertise is knowing when reinforcement learning is appropriate and when a simpler method would be better.

2.9 How to Read This Book with the Code

This book is meant to be read together with the maintained code in examples/. The goal is not to treat the code as a decorative appendix. The goal is to use the code to make the ideas operational.

The structure of the repository mirrors the conceptual arc of the book:

  1. examples/bandits/ covers the simplest exploration problems.
  2. examples/tabular/ covers dynamic programming, Monte Carlo prediction, TD learning, SARSA, and Q-learning.
  3. examples/approximation/ shows the bridge from tables to feature-based value approximation.
  4. examples/deep_rl/ contains the neural-network-based examples.

That structure matters for reading. If a chapter introduces a Bellman backup, you should be able to find a concrete loop in the corresponding script where that backup is computed. If a chapter distinguishes on-policy from off-policy learning, you should be able to locate the exact line where one method uses the actually chosen next action and another uses a greedy max.

This is also the point where honesty matters. Not every maintained example in this repository has the same status.

  1. Some examples are small, self-contained, and intended as direct principle-first references. The tabular scripts and the PyTorch DQN family mostly fit this category.
  2. Some examples are useful conceptual bridges but are more sensitive in practice. The function-approximation scripts belong here.
  3. Some policy-gradient examples use a mature third-party implementation rather than a local from-scratch reimplementation. That is a deliberate choice in this project, because for some actor-critic-family methods the stable engineering recipe matters as much as the headline formula.

The right way to use this book, then, is not to assume that every chapter points to code of the same kind. The right way is to ask, chapter by chapter:

  1. What is the core principle?
  2. Where does that principle appear in code?
  3. Is this code meant as a minimal transparent implementation, a practical baseline, or a maintained third-party-backed reference?

The next chapter introduces the mathematical framework that organizes these ideas: the Markov decision process.
