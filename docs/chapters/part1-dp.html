5  Dynamic Programming

Dynamic programming is one of the cleanest entry points into reinforcement learning. It is not the most realistic setting for large-scale applications, because it assumes access to the full environment model. But it provides an ideal laboratory for understanding value functions, Bellman equations, and policy improvement.

In an MDP, if we know the transition dynamics and reward function, then we can often compute value functions by repeated updates rather than by pure trial-and-error interaction. This is the basic idea of dynamic programming.

5.1 Why Study Dynamic Programming?

Dynamic programming matters for three reasons.

  1. It makes the Bellman equations operational.
  2. It separates evaluation from improvement very clearly.
  3. It provides the conceptual foundation for many later algorithms.

Even when we cannot use dynamic programming directly, its logic remains everywhere in reinforcement learning. Temporal-difference learning uses Bellman-style targets. Q-learning uses an optimality backup that mirrors dynamic programming. DQN approximates those same updates with a neural network.

So while dynamic programming is not the final practical answer, it is one of the best conceptual starting points.

If you remember one thing from this chapter, it should be this:

Dynamic programming is where Bellman equations stop being definitions and start becoming procedures.

5.2 Policy Evaluation

Suppose we are given a policy \(\pi\). Policy evaluation asks a single question:

How good is this policy?

More precisely, we want to compute the value function \(V^\pi(s)\) for each state. From the Bellman expectation equation, we know that:

\[ V^\pi(s) = \sum_a \pi(a \mid s)\sum_{s',r} P(s', r \mid s, a)\left[r + \gamma V^\pi(s')\right] \]

This is a system of equations, one for each state. In small problems, we may solve it directly. More often, we use iterative policy evaluation:

\[ V_{k+1}(s) \leftarrow \sum_a \pi(a \mid s)\sum_{s',r} P(s', r \mid s, a)\left[r + \gamma V_k(s')\right] \]

The idea is simple. Start with an initial guess for the value function, often all zeros. Then repeatedly apply the Bellman update until the values stabilize.

Each update uses the current estimate \(V_k\) to produce a better estimate \(V_{k+1}\). Under standard conditions, this process converges to the true value function of the policy.

In plain language, policy evaluation means:

Keep asking “if I keep behaving this way, how good is each state?” until the answers stop changing much.

In the maintained code for this book, this idea appears directly in:

  1. examples/tabular/policy_evaluation.py

The key function is evaluate_deterministic_policy(...). It does exactly what the equation above says: for each non-terminal state, it loops over actions and next states, computes the Bellman expectation backup, and repeats until the largest change across states falls below a tolerance threshold.

This is worth pausing over. In many books, iterative policy evaluation is presented only as a formula. In code, the crucial operational detail is the stopping condition. The implementation does not stop because it knows the true answer. It stops because successive sweeps stop changing the value function by much.

5.3 A Beginner Walkthrough of the Bellman Sweep

If you want to see policy evaluation as executable logic, the heart of the implementation is this pattern:

for state in grid.all_states():
    ...
    for next_state in grid.all_states():
        new_value += action_prob * transition_prob * (reward + gamma * values[next_state])

That is the Bellman equation turned into a loop.

For beginners, the key insight is not the nested loop structure itself. It is the meaning:

  1. consider a state,
  2. consider what action the policy takes there,
  3. consider where that action can lead,
  4. average immediate reward plus discounted next-state value.

Once that meaning is clear, dynamic programming stops feeling mysterious.

5.4 Policy Improvement

Once we know how good a policy is, the next natural step is to improve it.

Suppose we already have \(V^\pi\). For each state, we can ask which action gives the highest expected return if we act once and then continue according to the old policy. That leads to a greedy improvement step:

\[ \pi'(s) = \arg\max_a \sum_{s',r} P(s', r \mid s, a)\left[r + \gamma V^\pi(s')\right] \]

This gives a new policy \(\pi'\). A key result in reinforcement learning is the policy improvement theorem, which says that this greedy policy is at least as good as the original policy under suitable conditions.

This is the basic pattern:

  1. Evaluate the current policy.
  2. Improve the policy greedily with respect to the current value estimates.
  3. Repeat.

That alternating structure is the heart of policy iteration.

This is also the first clean example in the book of a general RL pattern:

  1. estimate,
  2. improve,
  3. repeat.

5.5 Policy Iteration

Policy iteration combines policy evaluation and policy improvement into a loop.

The procedure is:

  1. Start with an initial policy.
  2. Evaluate the policy to obtain \(V^\pi\).
  3. Improve the policy greedily.
  4. If the policy no longer changes, stop. Otherwise, repeat.

This method is elegant because it mirrors how we often think about learning in general. First understand the consequences of the current behavior, then update the behavior based on that understanding.

In small finite MDPs, policy iteration can converge surprisingly quickly. It is especially useful for grid worlds and textbook environments, where the transition model is available and easy to inspect.

The maintained implementation is:

  1. examples/tabular/policy_iteration.py

There are two functions worth reading together:

  1. run_policy_iteration(...)
  2. improve_policy(...)

The first orchestrates the outer loop. The second is where the greedy improvement step actually happens. If you want to see the policy improvement theorem in code form, this file is one of the cleanest places in the project to do it. The code evaluates the current policy, then for each state compares all available actions under the current value estimates, and replaces the old action with the best one.

For many beginners, this is the first moment when reinforcement learning starts to feel algorithmic rather than purely definitional. The loop is no longer just “compute values.” It is:

  1. understand the current policy,
  2. then replace it with a better one.

5.6 Value Iteration

Policy iteration is conceptually clean, but full policy evaluation can be expensive. Value iteration compresses the evaluation and improvement steps into a single update.

Instead of fully evaluating the current policy, value iteration directly applies the Bellman optimality update:

\[ V_{k+1}(s) \leftarrow \max_a \sum_{s',r} P(s', r \mid s, a)\left[r + \gamma V_k(s')\right] \]

This update says: if the goal is optimal behavior, then at each iteration we should back up the best action immediately.

After the value function converges, we can recover an optimal policy by choosing the greedy action in each state:

\[ \pi^*(s) = \arg\max_a \sum_{s',r} P(s', r \mid s, a)\left[r + \gamma V^*(s')\right] \]

Value iteration is often simpler to implement than policy iteration, and it highlights the role of the Bellman optimality equation very clearly.

The maintained implementation is:

  1. examples/tabular/value_iteration.py

Compared with policy iteration, the important conceptual difference in code is this: the update takes a max inside the sweep itself. In other words, policy improvement is no longer a separate outer step. The Bellman optimality operator is applied directly during value updates.

That difference is visible in run_value_iteration(...), where each state update computes the value of every possible action and immediately keeps only the largest one. After convergence, the script performs one more greedy pass to extract the policy explicitly.

In plain language, value iteration says:

Do not fully evaluate the current policy first. Just keep pushing every state toward the best-looking action immediately.

5.7 Common Beginner Confusions About Dynamic Programming

5.7.1 “If dynamic programming needs the whole model, why study it?”

Because it makes the Bellman logic transparent. Later methods often remove the full-model assumption, but they keep the same backup structure.

5.7.2 “What is the real difference between policy iteration and value iteration?”

Policy iteration alternates between evaluating a policy and improving it. Value iteration folds improvement directly into the value update with a max. That is the core difference to watch for in code.

5.8 Synchronous and In-Place Updates

There are several ways to implement dynamic programming updates. In synchronous updates, every state is updated using values from the previous sweep. In in-place updates, the algorithm overwrites values as soon as they are computed, allowing later states in the same sweep to use fresher information.

This detail may seem minor, but it affects convergence speed. In-place updates often propagate useful information faster. More advanced planning methods build on this idea by choosing which states to update in more strategic ways.

The current tabular dynamic-programming examples in this project use straightforward sweep-based updates. That is a good choice for a foundation text. It keeps the Bellman logic visible and avoids burying the main idea under optimization tricks.

5.9 The Limits of Dynamic Programming

Dynamic programming is powerful, but its assumptions are strong.

First, it requires a model of the environment. In many real-world problems, we do not know the transition probabilities or reward function exactly.

Second, it assumes the state space is manageable. If the number of states is huge, storing and updating a value for every state becomes infeasible.

Third, it depends on repeated sweeps over the entire state space. This is difficult in high-dimensional or continuous domains.

These limitations explain why dynamic programming is not the dominant method in modern large-scale reinforcement learning. Still, the logic of Bellman backups and iterative improvement survives in almost every major algorithm family.

5.10 From Planning to Learning

Dynamic programming is often described as a planning method rather than a learning method. That distinction is useful:

  1. Planning uses a known model to compute good behavior.
  2. Learning uses experience to estimate what the model or values should be.

The next chapter makes this transition explicit. We will drop the assumption of a known model and ask how an agent can estimate value functions directly from sampled episodes and transitions. That leads to Monte Carlo methods and temporal-difference learning.

For this reason, dynamic programming should be read here less as a practical recipe and more as a conceptual template. The exact same Bellman logic that appears in these grid world scripts later reappears in TD learning, Q-learning, DQN, and even actor-critic critics. The implementations become more approximate, more statistical, and more fragile, but the underlying backup idea does not disappear.
