9  Deep Q-Networks

Deep Q-Networks, usually abbreviated DQN, mark one of the most influential turning points in reinforcement learning. The core idea is easy to state:

Use a neural network to approximate the Q-function in a Q-learning-style algorithm.

The reason this mattered is that tabular Q-learning cannot handle raw, high-dimensional observations such as images. DQN showed that with the right stabilizing techniques, value-based reinforcement learning could scale far beyond small tabular environments.

9.1 From Q-Learning to DQN

Recall the tabular Q-learning update:

\[ Q(S_t, A_t) \leftarrow Q(S_t, A_t) + \alpha \left[R_{t+1} + \gamma \max_a Q(S_{t+1}, a) - Q(S_t, A_t)\right] \]

DQN keeps the same target structure, but replaces the table with a neural network:

\[ Q(s, a; \theta) \]

where \(\theta\) denotes the network parameters.

Instead of directly overwriting a table entry, we now train the network so that its prediction \(Q(S_t, A_t; \theta)\) moves toward a target value.

This is often written as:

\[ y_t = R_{t+1} + \gamma \max_{a'} Q(S_{t+1}, a'; \theta^-) \]

where \(\theta^-\) are parameters of a separate target network. We will explain that choice shortly.

Training then minimizes a loss such as:

\[ \mathcal{L}(\theta) = \mathbb{E}\left[\left(y_t - Q(S_t, A_t; \theta)\right)^2\right] \]

This looks close to supervised learning, but the similarity is misleading. In supervised learning, targets are usually fixed labels from a dataset. In DQN, the targets are themselves generated from the agent’s evolving estimates.

9.2 Why Naive Deep Q-Learning Is Unstable

If we simply plug a neural network into Q-learning, training can become highly unstable.

There are several reasons:

  1. Consecutive samples are strongly correlated.
  2. The target depends on the same network that is being updated.
  3. The method is off-policy.
  4. The updates rely on bootstrapping.

This creates a feedback loop. The network predicts values, those predictions define the targets, and those targets immediately reshape the network again. Without extra care, the process can oscillate or diverge.

That is why DQN should be understood not only as a neural version of Q-learning, but as a stabilization package for deep value learning.

9.3 Experience Replay

One of the most important DQN ideas is the replay buffer.

Instead of updating the network only from the newest transition, the agent stores transitions such as:

\[ (s, a, r, s', d) \]

in a memory buffer. During training, it samples minibatches from this buffer.

This has several benefits:

  1. It breaks the strong temporal correlation in consecutive experience.
  2. It reuses past data more efficiently.
  3. It makes the training distribution smoother and more diverse.

Replay turns reinforcement learning into something closer to stochastic optimization over a growing dataset. That shift is one reason deep value learning became much more workable.

Historically, even simple replay buffer implementations already showed the basic advantage. Modern systems may add prioritized sampling, larger buffers, or more careful data management, but the fundamental insight remains the same.

9.4 Target Networks

The second major stabilization idea is the target network.

If the same parameters are used both to predict the current Q-value and to compute the target, then the optimization target changes too quickly. The system is trying to chase its own moving predictions.

DQN reduces this problem by maintaining a second set of parameters, often written as \(\theta^-\), which are copied from the main network only periodically or updated more slowly.

The target then becomes:

\[ y_t = R_{t+1} + \gamma \max_{a'} Q(S_{t+1}, a'; \theta^-) \]

This simple idea makes a large practical difference. The target becomes more stable, which improves learning behavior.

Older implementations often copy the target network every fixed number of steps. Many modern methods still use that basic pattern, though details vary by algorithm and codebase.

9.5 The DQN Training Loop

At a high level, DQN training follows a repeated loop:

  1. Observe the current state.
  2. Choose an action, often with \(\epsilon\)-greedy exploration.
  3. Step the environment and collect the next transition.
  4. Store the transition in replay memory.
  5. Sample a minibatch from replay.
  6. Build bootstrap targets with the target network.
  7. Update the online Q-network by gradient descent.
  8. Periodically update the target network.

This loop is conceptually simple once written out, but each component plays a role in keeping learning stable enough to proceed.

In the maintained code for this project, the main reference implementation is:

  1. examples/deep_rl/dqn_cartpole.py

That file is intentionally small enough to be read end to end. If you want to connect the abstract DQN loop to concrete code, the main pieces are:

  1. ReplayBuffer, which stores transitions.
  2. QNetwork, which maps observations to one Q-value per action.
  3. select_action(...), which implements \(\epsilon\)-greedy behavior.
  4. optimize(...), which builds the bootstrap target and performs the gradient step.
  5. evaluate_policy(...), which separates exploratory training behavior from greedy evaluation.

This chapter is easier to internalize if you treat those names as the executable version of the pseudocode.

9.6 A Beginner Walkthrough of the Actual Code

If you are reading DQN for the first time, do not start by trying to understand the whole file at once. Read examples/deep_rl/dqn_cartpole.py in four passes.

9.6.1 Pass 1: What is being stored?

First find ReplayBuffer. Its entire job is to keep past transitions so training does not depend only on the newest step.

class ReplayBuffer:
    def add(self, state, action, reward, next_state, done):
        self.buffer.append((state, action, reward, next_state, done))

    def sample(self, batch_size):
        batch = random.sample(self.buffer, batch_size)
        states, actions, rewards, next_states, dones = map(np.array, zip(*batch))
        return states, actions, rewards, next_states, dones

The important beginner point is not the container type. The important point is that DQN learns from sampled old experience rather than only from the most recent transition.

9.6.2 Pass 2: What is the model predicting?

Then find QNetwork.

class QNetwork(nn.Module):
    def __init__(self, observation_dim, action_dim, hidden_dim):
        self.net = nn.Sequential(
            nn.Linear(observation_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, action_dim),
        )

For CartPole, the network takes a state vector and outputs one Q-value per action. That means the model is not asked “what is the next action?” It is asked:

What are my current value estimates for each action in this state?

That distinction matters. DQN is still a value-based method. The policy comes later from choosing the action with the largest predicted value.

9.6.3 Pass 3: Where is the Bellman target?

The heart of the algorithm is in optimize(...).

current_q_values = q_network(state_tensor).gather(1, action_tensor).squeeze(1)

with torch.no_grad():
    next_q_values = target_network(next_state_tensor).max(dim=1).values
    targets = reward_tensor + config.gamma * (1.0 - done_tensor) * next_q_values

This is the place where the chapter equation becomes code.

  1. q_network(state_tensor) predicts values for all actions now.
  2. .gather(...) picks the value of the action that was actually taken.
  3. target_network(next_state_tensor).max(...) estimates the best next action value.
  4. reward + gamma * next_value builds the bootstrap target.

If a beginner understands only those four lines, the core of DQN is already in view.

9.6.4 Pass 4: Where is the learning loop?

Finally look at train_dqn_cartpole(...). Inside the main loop, the file does the same cycle repeatedly:

  1. choose an action with epsilon-greedy behavior,
  2. step the environment,
  3. store the transition,
  4. run optimize(...),
  5. periodically copy parameters into the target network.

That is the operational meaning of DQN. Everything else in the file supports one of those steps.

9.7 Common Beginner Confusions About DQN

Several confusions are especially common here, so it helps to answer them directly.

9.7.1 “If the network outputs an action, why do we still need epsilon-greedy?”

The network does not output an action. It outputs estimated values for actions. The agent still needs a behavior rule for choosing actions during training. In this implementation, that rule is \(\epsilon\)-greedy: usually choose the current best-looking action, but sometimes explore.

9.7.2 “Why do we need two networks if they have the same architecture?”

Because they play different roles.

  1. The online network is the one being updated right now.
  2. The target network is a slower-moving reference used to build the target.

The point is not architectural diversity. The point is target stability.

9.7.3 “Why does DQN feel much harder than tabular Q-learning even though the formula looks similar?”

Because the hard part is no longer only the formula. The hard part is the interaction among:

  1. bootstrapping,
  2. off-policy learning,
  3. shared function approximation.

In tabular Q-learning, that interaction is much easier to survive. In deep RL, it can destabilize training quickly.

9.8 Input Representation and Atari

DQN became famous because of its Atari results. In that setting, the network processed stacks of image frames rather than low-dimensional hand-crafted state features.

That achievement is historically important because it demonstrated end-to-end value learning from high-dimensional sensory input. The network did not rely on a manually designed state table. It learned directly from a representation much closer to raw observation.

At the same time, it is important not to mythologize the result too much. DQN also depended on careful preprocessing, frame stacking, reward clipping, and other design choices. In reinforcement learning, seemingly small implementation decisions often matter a great deal.

9.9 What Old DQN Code Still Teaches

Older DQN implementations often use TensorFlow 1.x and earlier Gym APIs. As software artifacts, they are dated, but they still teach several enduring lessons:

  1. The Q-network predicts one value per action.
  2. A replay buffer stores transitions for later sampling.
  3. A separate target network stabilizes bootstrap targets.
  4. Exploration is usually handled by an \(\epsilon\)-greedy policy.

Those are the algorithmic ideas worth carrying forward. The exact framework syntax, environment API, and engineering style should be treated as historical rather than current.

At the same time, the maintained examples in this project now include a modern PyTorch CartPole DQN implementation and a matching Double DQN variant. Those files are intentionally smaller and cleaner than the legacy scripts so that readers can trace the training loop, replay buffer usage, and target-network logic without needing to decode TensorFlow 1.x graph code first.

The maintained value-based path in this repository now consists of:

  1. examples/deep_rl/dqn_cartpole.py
  2. examples/deep_rl/double_dqn_cartpole.py
  3. examples/deep_rl/dueling_dqn_cartpole.py
  4. examples/deep_rl/prioritized_dqn_cartpole.py
  5. examples/deep_rl/rainbowish_cartpole.py

These files are not identical in status.

  1. The first two are the clearest starting points if the goal is to understand the core algorithm.
  2. The next two add specific stabilization or representation ideas on top of that base.
  3. The last one is intentionally described as “Rainbow-ish” rather than full Rainbow, because it combines several ideas without claiming to be a complete reproduction of the full Rainbow recipe.

9.10 Main Weaknesses of Basic DQN

Basic DQN was a breakthrough, but it has important limitations.

First, it is mainly designed for discrete action spaces. The max over actions is manageable when the action set is small, but it becomes difficult in continuous control.

Second, it can overestimate action values because the same estimates are used inside a max operator. This issue motivated later methods such as Double DQN.

Third, performance can be highly sensitive to reward scaling, exploration schedules, replay design, target update frequency, and network architecture.

Fourth, DQN is sample-hungry. Learning strong behavior can require many environment interactions.

These limitations do not make DQN unimportant. On the contrary, they explain why so much later research built directly on it.

9.11 Beyond Basic DQN

Several influential methods can be understood as refinements of DQN:

  1. Double DQN reduces overestimation bias.
  2. Dueling networks separate value and advantage structure.
  3. Prioritized replay samples transitions more selectively.
  4. Distributional methods model return distributions rather than only expectations.

These extensions improved performance and stability in many settings, but the basic DQN structure remains a reference point for value-based deep reinforcement learning.

In the code for this book, the DQN path is now represented by:

  1. examples/deep_rl/dqn_cartpole.py
  2. examples/deep_rl/double_dqn_cartpole.py
  3. examples/deep_rl/dueling_dqn_cartpole.py
  4. examples/deep_rl/prioritized_dqn_cartpole.py
  5. examples/deep_rl/rainbowish_cartpole.py

That pairing is useful because it makes the target-construction difference concrete. Basic DQN uses the target network both to select and evaluate the next greedy action. Double DQN instead uses the online network to choose the next action and the target network to evaluate that chosen action.

The broader family is also useful pedagogically because it shows how modern value-based reinforcement learning usually evolves in practice. Rarely does a project jump directly from tabular Q-learning to a fully elaborate agent. More often, it accumulates stabilizing ingredients one by one: first replay and target networks, then improved targets, then architectural changes, then improved sampling schemes.

9.12 Why DQN Still Matters

DQN still matters for at least three reasons.

First, it is the clearest bridge from classical Q-learning to modern deep reinforcement learning.

Second, it illustrates the practical consequences of the deadly triad: bootstrapping, off-policy learning, and function approximation. The replay buffer and target network are direct responses to those problems.

Third, it teaches a broader lesson that appears throughout DRL:

Good reinforcement learning algorithms are often not just mathematical objectives. They are objectives plus stabilization mechanisms.
