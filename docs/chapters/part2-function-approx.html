7  From Tabular Methods to Function Approximation

By this point, we have seen several core reinforcement learning ideas in their tabular form. We estimated values, improved policies, compared Monte Carlo and temporal-difference learning, and studied classical control methods such as SARSA and Q-learning.

These methods are conceptually important, but they face a serious limitation:

They do not scale well when the state space becomes large, high-dimensional, or continuous.

This chapter explains why that limitation appears, why function approximation becomes necessary, and how this transition leads directly into deep reinforcement learning.

7.1 Why Tabular Methods Stop Scaling

In a tabular method, the agent stores a separate value for each state or each state-action pair. This works well in small environments such as grid worlds, toy games, or discretized control tasks.

But many realistic problems do not have small state spaces.

Consider a robotic control task with continuous position and velocity variables. Consider a video game where the state is an image. Consider any problem where the observation includes dozens or hundreds of features. In these settings, we cannot expect to visit every possible state enough times to learn a reliable table entry.

There are several related problems:

  1. The number of states may be enormous.
  2. The state space may be continuous rather than discrete.
  3. Similar states should share information, but a lookup table cannot generalize.

This last point is especially important. In tabular learning, each state is treated as isolated. Learning that one state is valuable tells us nothing directly about nearby or similar states. That becomes extremely inefficient in large environments.

7.2 Generalization as the Key Motivation

Function approximation solves this problem by replacing explicit tables with parameterized models.

Instead of storing one independent value for each state, we define a function such as:

\[ \hat{V}(s; w) \]

or

\[ \hat{Q}(s, a; w) \]

where \(w\) denotes learnable parameters.

Now the agent no longer memorizes values one state at a time. Instead, it learns a mapping from states or state-action pairs to predicted values. Similar inputs can produce similar outputs, so information can generalize across related situations.

This shift changes the entire learning problem. The goal is no longer just to fill in a table. The goal is to learn a useful approximation of the value function or policy over a large input space.

7.3 Linear Function Approximation

A natural first step is linear approximation. Suppose we extract a feature vector \(x(s)\) from state \(s\). Then we may write:

\[ \hat{V}(s; w) = w^\top x(s) \]

For action values, we may use features of state-action pairs:

\[ \hat{Q}(s, a; w) = w^\top x(s, a) \]

This looks simple, but it already changes the learning behavior in an important way. Updating the parameter vector \(w\) changes the prediction for many states at once. This is the source of generalization.

Older reinforcement learning examples often illustrate this step with hand-designed features, radial basis function expansions, or other shallow approximators. Those examples remain useful because they expose the transition from tabular storage to shared parameterization very clearly.

7.4 A Conceptual Bridge from Old Examples

In this repository, the maintained bridge examples now live in:

  1. examples/approximation/cartpole_linear_q.py
  2. examples/approximation/mountaincar_linear_q.py

These scripts use explicit feature engineering together with a shallow linear model to approximate action values. Although they are much smaller than modern deep RL systems, their conceptual structure is exactly the bridge we need:

  1. Represent a state-action pair as a feature vector.
  2. Predict a scalar value from that representation.
  3. Build a bootstrap target from reward plus an estimate of the next state’s best action value.
  4. Update the model parameters toward that target.

That structure is the real bridge. Deep reinforcement learning keeps the same high-level pattern, but replaces hand-designed features and shallow models with neural networks.

At the same time, these maintained examples should not be interpreted as equally strong engineering baselines. The CartPole approximation example is currently the more stable of the two. The MountainCar version is useful conceptually, but remains more sensitive to feature choice and hyperparameters.

This distinction is important for how this chapter should be read. The role of these files is not mainly to provide the strongest benchmark results in the repository. Their role is to let the reader see, in a small and inspectable form, what changes when a lookup table becomes a parameterized function:

  1. the value estimate becomes a prediction from features,
  2. a single update changes many states at once,
  3. and stability becomes partly a representation problem rather than only a sampling problem.

7.5 The Promise and Risk of Approximation

Function approximation gives us scalability and generalization, but it also introduces new difficulties.

In a table, an update only changes one entry. With approximation, a single update can affect many predictions at once. This means generalization can help, but it can also cause interference. Improving one region of the state space may accidentally damage estimates elsewhere.

This leads to a new tension:

  1. Shared parameters make learning more efficient.
  2. Shared parameters also make learning less local and potentially less stable.

That trade-off becomes one of the defining difficulties of deep reinforcement learning.

7.6 Prediction with Function Approximation

The simplest way to combine temporal-difference learning with approximation is to keep the TD target but optimize model parameters instead of table entries.

For example, with state-value prediction we may use a target:

\[ y_t = R_{t+1} + \gamma \hat{V}(S_{t+1}; w) \]

and then update parameters to reduce the difference between the current prediction and that target.

In a linear model, the update may resemble stochastic gradient descent. In a neural network, the same principle applies, but the function class is much richer.

The crucial idea is that the Bellman logic remains. The representation changes, but the reinforcement learning objective is still driven by returns, value estimates, and recursive consistency.

7.7 Control with Function Approximation

We can extend the same idea to action-value control. Instead of storing a tabular Q-function, we approximate:

\[ \hat{Q}(s, a; w) \]

Then we can construct targets that resemble Q-learning:

\[ y_t = R_{t+1} + \gamma \max_{a'} \hat{Q}(S_{t+1}, a'; w) \]

and update parameters so that \(\hat{Q}(S_t, A_t; w)\) moves toward \(y_t\).

Conceptually, this is a direct continuation of tabular Q-learning. But in practice, the combination of three ingredients can be unstable:

  1. Bootstrapping.
  2. Off-policy learning.
  3. Function approximation.

This combination is so important that it is sometimes called the “deadly triad” in reinforcement learning. It explains why naive Q-learning with nonlinear approximators can behave poorly.

7.8 Why Neural Networks Enter the Picture

If linear approximation already offers generalization, why do we need deep learning?

The answer is representation power. Linear models depend heavily on feature design. If the right features are not provided, the model may fail even when the task is learnable. In complex domains such as images, raw sensory streams, or large structured observations, manual feature design is often insufficient or too expensive.

Neural networks allow the agent to learn representations automatically. Rather than hand-crafting features for every state, we train a network to extract patterns that are useful for predicting values or selecting actions.

This is the core promise of deep reinforcement learning:

Learn representations and decision-making signals together.

But this promise comes with a cost. Neural networks are flexible, high-capacity function approximators, which also makes them harder to stabilize under reinforcement learning objectives.

7.9 The Difference Between Historical and Modern Practice

Older code examples often demonstrate the right algorithmic skeleton but use tools and assumptions that no longer match common modern workflows. You may see:

  1. TensorFlow 1.x graph construction.
  2. Older Gym APIs.
  3. Minimal replay buffer implementations.
  4. Limited experiment tracking.
  5. Little attention to evaluation across random seeds.

These examples are still worth reading, especially for conceptual clarity. But a modern practitioner should separate the enduring ideas from the outdated tooling.

The enduring ideas include:

  1. Temporal-difference targets.
  2. Approximate value functions.
  3. Exploration strategies.
  4. Target computation for control.

The tooling details, however, should be updated when building serious systems today.

The same caution applies inside this project. The maintained approximation examples are honest bridge examples, not claims that shallow feature engineering is the best practical solution for these tasks. Their real value is explanatory:

  1. they preserve the Bellman-style target clearly,
  2. they show where handcrafted representation enters,
  3. and they expose, at a small scale, why approximation makes learning more sensitive.

7.10 The Road to DQN

Deep Q-Networks, or DQN, were historically important because they showed that Q-learning-style updates could be combined with deep neural networks to learn directly from high-dimensional inputs such as Atari frames.

This was not merely “Q-learning plus a larger model.” Several stabilizing ideas were needed to make training work at all, especially:

  1. Experience replay.
  2. Target networks.
  3. Careful preprocessing and training design.

The next chapter studies DQN in detail and explains why those additions were so important.
